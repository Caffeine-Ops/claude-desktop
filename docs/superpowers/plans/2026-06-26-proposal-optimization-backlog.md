# 「写方案」功能 · 产品优化 Backlog

> 产出自 office-hours / PM 视角分析（2026-06-26）。北极星指标：**方案可直接交付率**——售前/解决方案人员少改甚至不改就能把生成稿发给客户的比例。
>
> 每条优化都要能回答一句：它把可直接交付率往上推了，还是只是让产品看起来更全？
>
> 现状一句话：**骨架完整，卡在「能用 → 敢交付」之间。**

---

## 度量层

### M-0 · 埋两个核心指标

**问题**：当前优先级是假设驱动的，只有「AI 质量差」一句口述，没有真实数据。无法判断可交付率到底卡在内容还是排版。

**改动**：
- **可交付率代理指标**：导出后记录「导出时 markdown」与「用户最终发出版本」差异不可得时，退而求其次——记录从生成完成到点导出之间，用户对 sections 的编辑次数/改动字数比例（store 里 `updateSection` 调用已可埋点）。
- **引用准确度**：抽样任务——对一份方案的每条「（据《X》）」回原文核对，统计编造率与引错率。先人工跑一次建立基线，后续可由 P0-1 自动产出。

**验收标准**：
- [ ] 每次导出在本地（不外传）落一条记录：sectionCount、累计编辑字数、各 kind 段落数。
- [ ] 有一份手工核对的基线报告：编造率 X%、引错率 Y%，覆盖 ≥1 份真实方案。
- [ ] 文档写明这两个数如何反推 backlog 重排。

**工作量**：S（~1d / ~1.5h）

## P0 · 信任线（决定「敢不敢真发给客户」）

### P0-1 · 落地校验 + 覆盖度红灯

**问题**：
- 来源标注「（据《X》）」是 AI **写完才贴**的（`proposalPrompt.ts:75`），没有任何东西校验那句话真在 X 里。编造无法被系统拦住——而这是要交给客户做采购决策的文档（`proposalPrompt.ts:61` 自陈「任何编造都会造成实质损害」）。
- 用户看不出某章是「真有料」还是「AI 兜底糊的」，被迫逐句人肉核对。

**改动**：
- **落地校验回合**：每章正文生成后（哨兵块闭合时），解析其中所有「（据《X》）」引用，对镜像文件 X 做存在性 + 关键句匹配核对（关键句级模糊匹配即可，不必逐字）。不匹配的句子在 `ProposalPaper` 里就地标红「⚠️ 这句在《X》里没找到，疑似编造」。
- **覆盖度可见化**：每章头部显示「本章引用 N 个文件」，N=0 时红灯。
- 校验在主进程做（能读镜像文件），结果随 section 元数据回传 renderer。

**验收标准**：
- [ ] 故意让 AI 编一句无出处内容 → 该句在编辑区被标红。
- [ ] 引用了真实存在的文件且句子可溯源 → 不标红。
- [ ] 引用了不存在的文件名 → 标红并提示文件未找到。
- [ ] 某章 0 引用 → 章头红灯。
- [ ] 校验失败/镜像读不到时降级为「未校验」灰标，绝不阻塞生成或导出。
- [ ] 校验全程不修改正文内容，只标注。

**工作量**：M（~3d / ~4h）　**依赖**：无（与 M-0 的引用准确度自动化天然合一）

---

### P0-2 · 内容级召回注入（治检索盲区）

**问题**：系统**没有真正的检索**。`proposalPrompt.ts:55` 只给 AI 文件**标题清单**（最多 50 个，`MAX_FILES_PER_PRODUCT`），AI 从没看过内容就靠标题猜该 Read 哪个。料埋在标题不显眼的文件里就引不到——这是「有料却没引到 / 空泛 / 反复追问才逼出来」的共同根因。

**改动**：
- 在 `kb-index` 旁建一层**内容级搜索**（mirror 已是纯文本，BM25/grep 级关键词召回即可起步，不强求 embedding）。
- 每章写作前，用该章标题 + 用户需求做召回，把命中的**真实段落原文**注入这一轮上下文，而不是只给文件标题让 AI 自己猜。
- 召回范围限定在已 seed 的产品目录内，保持「只用知识库」底线。

**验收标准**：
- [ ] 给定一个标题不含关键词、但正文含关键料的文件 → 召回能命中并注入。
- [ ] 注入的段落带文件来源，能被 P0-1 校验链复用。
- [ ] 召回为空时明确告知 AI「未召回到相关原文」，AI 据此走「资料缺失」而非兜底编造。
- [ ] 召回耗时不显著拖慢首字（建立索引一次性，查询亚秒级）。
- [ ] 对比基线：同一组方案需求，引错率/空泛率较 M-0 基线下降（具体阈值由基线定）。

**工作量**：L（~5d / ~1d）　**依赖**：建议 M-0 先行以便量化收益。

## P1 · 控制力（决定「在这儿改稿，还是导出去 Word 改」）

### P1-1 · 段落级轻量编辑指令

**问题**：改一句要重发整轮 AI turn。没有「重写这段 / 换个说法 / 这章再详细些 / 这句删掉」的轻量入口。编辑摩擦大，用户倾向导出后逃去 Word，工具退化成一次性生成器。

**改动**：
- `ProposalPaper` 段落悬停工具条增加轻量指令入口：选中一段 →「重写 / 扩写 / 精简 / 更正式」等预设指令。
- 这些指令构造一条聚焦该段的 AI 消息（带原段落 + 指令），回流只替换该段，不重生成整章。
- 复用现有哨兵机制：让 AI 用对应 kind 哨兵只回这一段。

**验收标准**：
- [ ] 选中一段点「精简」→ 只有该段被替换，其余段落不动。
- [ ] 指令结果同样走 P0-1 校验（不能越改越编）。
- [ ] 指令失败/AI 跑偏时原段落不丢失（先留旧、确认后替换）。
- [ ] 段落级编辑不破坏 kind 连续性与分节。

**工作量**：M（~3d / ~4h）

---

### P1-2 · 可纠正产品集

**问题**：产品匹配零命中即 `seeded` 固定（`proposal.ts`，`FusionRuntimeProvider.tsx:524`），中途发现匹配错了或漏了产品，没有「重选/加产品」入口，只能整个重开。

**改动**：
- 工作台提供「编辑产品集」入口，列出 KB 全部可选产品（`proposalProductScopes` 已有数据），允许增删。
- 改动后更新 store products，并让下一轮 spawn/注入用新产品集（注意 warm-spawn 注入路径，`engine.ts`）。

**验收标准**：
- [ ] 中途加一个产品 → 后续章节召回/检索范围包含它。
- [ ] 删一个产品 → 后续不再读该产品目录。
- [ ] 已生成的正文不受影响（只影响后续生成）。
- [ ] 零命中场景下用户能手动指定产品集，不必重开方案。

**工作量**：M（~2d / ~3h）

---

### P1-3 · 解锁跨章重排 + 修截断 kind

**问题**：
- `moveSection` 只允许同 kind 内交换（`proposal.ts` 不变量，为保 kind 连续/分页正确），用户无法把正文某章整体往前挪。
- 流被截断的残块带着起始哨兵的 kind，预览/导出会错位。

**改动**：
- 允许正文章节（content kind 内）跨位置重排，同时保证导出分节按最终顺序重算（解耦「移动约束」与「分节正确性」——后者应由导出时按 kind 分组保证，而非靠移动约束兜底）。
- 截断残块在 UI 明确标记「未完成片段」，且不参与分节（现已不参与，补 UI 提示），提供「重新生成此段」入口。

**验收标准**：
- [ ] 正文章节可任意重排，导出 docx 分节/分页仍正确。
- [ ] cover/toc/content 三大区段顺序仍受保护（不能把正文挪到封面前）。
- [ ] 截断残块在编辑区有醒目「未完成」标记，可一键续写/重生。

**工作量**：M（~2d / ~3h）

## P2 · 交付最后一公里（决定「专业感」）

### P2-1 · 品牌化导出（logo + 页眉页脚）

**问题**：样式三模板硬编码（`shared/proposalStyle.ts`），无公司 logo、页眉页脚、封面背景。B2B 售前方案的专业感一大半在品牌与排版，这块现在是裸的。销售必问。

**改动**：
- `ProposalStyleConfig` 扩展：logo 图片、页眉文字/图、页脚（页码 + 公司名）。
- `proposalDocx.ts` 在 docx 生成时注入页眉页脚与封面 logo。
- 样式弹窗（`ProposalStyleModal`）增加品牌配置项 + 实时预览。

**验收标准**：
- [ ] 上传 logo → 封面与/或页眉显示，导出 docx 中存在。
- [ ] 页脚显示页码 + 公司名，分节后页码连续正确。
- [ ] 预览与导出逐像素一致（维持单一 `markdownToDocxBuffer` 入口的不变量）。
- [ ] 不配置品牌时回退当前裸样式，不破坏现有方案。

**工作量**：M（~3d / ~4h）

---

### P2-2 · PDF 导出

**问题**：只有 md/docx（`proposalExport.ts` 的 FORMAT_META 白名单）。客户定稿交付常要 PDF。

**改动**：
- FORMAT_META 增加 `pdf`（白名单穷尽性守卫会在缺 key 处报错，单一真相源）。
- docx → PDF 走可靠转换路径（评估 libreoffice headless / docx-to-pdf 库；或先 docx 后由用户系统转，明确告知）。

**验收标准**：
- [ ] 导出 PDF 与 docx 排版一致（字体、分页、页眉页脚）。
- [ ] 转换失败有明确报错，不静默产出空/坏文件。
- [ ] 新增格式不破坏 md/docx 现有路径。

**工作量**：S（~1d / ~2h，若引入外部转换器另算）

---

### P2-3 · 样式模板可配置 / 自定义

**问题**：加模板要改 `proposalStyle.ts` 代码，运营/售前自己加不了。

**改动**：允许用户基于现有模板派生并保存自定义模板（字体/字号/缩进/品牌），持久化到 `proposalStyle` store（已跨会话存 localStorage）。

**验收标准**：
- [ ] 用户改完样式可「另存为模板」，下次方案可选。
- [ ] 自定义模板与内置模板共存、可删。
- [ ] 损坏的自定义模板回退默认，不崩。

**工作量**：M（~3d / ~4h）　**依赖**：P2-1（共用扩展后的 config 结构）

## P3 · 上手与健壮性

### P3-1 · 三阶段推进可发现性引导

**问题**：用户怎么知道要点「确认封面/确认目录」才推进？线性三阶段对新用户不直观，易卡在第一阶段。

**改动**：工作台显示当前阶段进度（封面 → 目录 → 正文），下一步动作用显式按钮 + 文案引导，而非埋在对话里。

**验收标准**：
- [ ] 任意时刻能看出「我在第几阶段、下一步该点什么」。
- [ ] 推进按钮文案明确（如「确认封面，生成目录」）。
- [ ] 阶段已由 `phase` 字段驱动，UI 与之一致。

**工作量**：S（~1d / ~2h）

---

### P3-2 · 「资料缺失」补料闭环

**问题**：AI 写「⚠️ 资料缺失」之后是断头路——用户怎么补料、补完怎么续，没有路径。

**改动**：把「资料缺失」标记结构化呈现，提供「我来补充这段料」入口（用户贴文本/指认文件），补完让 AI 续写该处。

**验收标准**：
- [ ] 资料缺失项在 UI 聚合可见，不淹没在对话里。
- [ ] 用户补料后能定点续写，不重生成整章。
- [ ] 补的料同样进 P0-1 校验来源链。

**工作量**：M（~2d / ~3h）　**依赖**：P0-2（召回链路）

---

### P3-3 · 草稿保存失败反馈

**问题**：`PROPOSAL_SAVE_DRAFT` 返回 `{ok:false}` 但 renderer 不检查（`register.ts:1078` 附近）。写盘失败（磁盘满/权限）用户无感，误以为已保存。

**改动**：renderer 检查保存结果，失败时给非阻塞提示（toast），并保留内存草稿不清。

**验收标准**：
- [ ] 模拟写盘失败 → 用户看到「草稿未保存」提示。
- [ ] 失败不阻塞会话切换（尽力而为精神不变）。
- [ ] 成功时不打扰（静默）。

**工作量**：S（~0.5d / ~1h）

---

### P3-4 · consumedDraftIds 重启隐患修复

**问题**：`consumedDraftIds` 刻意不持久化（`proposal.ts`），重启后从 transcript 重建。去重虽在抽取阶段兜住，但逻辑上仍有重复处理隐患。

**改动**：评估是否将 consumedDraftIds 纳入持久草稿记录，或在 transcript 重建路径补强去重断言（哪种更轻取哪种）。

**验收标准**：
- [ ] 重启后重开方案会话，已收录段落不重复 append。
- [ ] 不引入新的草稿膨胀。

**工作量**：S（~0.5d / ~1h）

---

## 诚实提醒

这份 backlog 的排序是**假设驱动**的，建立在代码全景图 + 一句「AI 质量差」之上。M-0 跑出真实的可交付率与引用准确度后，P0~P2 很可能重排——例如若可交付率其实卡在排版而非内容，P2-1 该提到 P0。**先做 M-0，让数据替你定优先级。**
