# 知识库驱动的方案写作功能 — 设计文档

- 日期：2026-06-23
- 状态：设计待评审
- 作者：kika + Claude
- 所在工程：`apps/desktop`（Electron 端，monorepo）

## 1. 背景与目标

福鑫数科是一家做医疗 AI 产品的公司，售前需要频繁写"建设方案/解决方案"
（典型产出见公司知识库里已有的成品 docx）。当前靠人工套模板写，慢且质量不稳。

本功能要在现有 **claude-desktop**（Electron 套壳 fusion-code 的聊天桌面应用）里，
加一个**知识库驱动的方案写作功能**：用户选一个产品 → AI 以对话向导方式，
**严格基于公司知识库**逐段写出方案草稿 → 用户轻量编辑 → 导出。

### 不可妥协的底线
> **实事求是，只用知识库里的内容写，绝不臆想、不编造。**
> 知识库里查不到的内容，必须明确标注"资料缺失"，而不是凭模型知识填补。

## 2. 关键决策（已与用户确认）

| # | 决策 | 选定方案 |
|---|------|---------|
| 1 | 模板形态 | **先写死在代码里**（章节骨架 + 产品线），自定义模板留进阶 |
| 2 | 向导交互 | **AI 对话驱动**（复用现有 chat，AI 问、用户答），非表单 |
| 3 | 知识库接入 | 严格基于知识库、强制来源标注；**预转换 + 建索引**（非按需转换） |
| 4 | 文本镜像位置 | 放 App **userData**，不污染原始资料 |
| 5 | MVP 导出格式 | 仅 **Markdown**，Word / PDF 等格式留进阶 |
| 6 | 索引抽取范围 | MVP 即抽**文字 + 媒体资产**（为进阶功能①铺路，避免 3GB 重转） |

## 3. 知识库现状（已探查）

- 路径（默认，做成可配置设置项）：
  `/Users/kika/Desktop/fusion方案资料/福鑫数科产品线资料库`
- 结构：**按产品线组织**，11 个一级目录（`01AI患者服务` … `11定制开发`），
  每条线下是具体产品（智能导诊、预问诊、DRG、病案质控…）。
  → 这套目录天然就是"模板分类"。
- 体量：312 个文件，约 **3.0 GB**。
- 文件类型：**192 docx + 45 xlsx + 37 pptx + 11 pdf** + 少量 txt/png/drawio/xmind/mp4。
  → 绝大多数是二进制 Office 文件，fusion-code 自带 `Read` 读不了，必须先转文本。
- 库内已含**成品方案 docx**，可同时作为"内容来源"与"结构范文"。

## 4. 总体架构

```
①索引层(离线一次性)              ②向导层(运行时)            ③文档层        ④导出
KB原始3GB ──markitdown──> 文本镜像  选产品 → AI对话向导        右侧可编辑     Markdown
docx/pptx/xlsx/pdf        (markdown) └ fusion-code 读镜像     Markdown草稿   (Word进阶)
                         +媒体资产    边检索边写、标来源
                         +index.json
```

四块职责单一、接口清晰：
- **①索引层**：把二进制资料变成 AI 能读的文本镜像 + 媒体资产 + 索引清单。离线、可增量。
- **②向导层**：复用现有 chat / ChatEngine，把会话 cwd 指向文本镜像，AI 对话驱动逐段写。
- **③文档层**：右侧轻量 Markdown 可编辑预览，承接 AI 草稿与用户改写。
- **④导出**：把最终 Markdown 落盘导出。

## 5. MVP 详细设计

### 5.1 ①索引层（新增核心基建）

- **转换工具**：主力 **markitdown**（微软开源，专为"喂文档给 LLM"做的
  docx/pptx/xlsx/pdf → Markdown 转换器，且能导出内嵌图片）；
  LibreOffice headless 作兜底（处理 markitdown 失败的文件）。
- **入口脚本**：`scripts/build-kb-index.ts`，批量转换，支持增量。
- **产物（放 App userData，例如 `<userData>/kb-index/`）**：
  - **文本镜像**：结构与原 KB 平行（`01AI患者服务/1_智能导诊系统/xxx.md`），
    体量从 3GB 降到几十 MB。
  - **媒体资产**：每个文档内嵌的图片/图表/drawio 抽到平行的 `assets/` 下，
    Markdown 里以 `![](...)` 引用。**MVP 只抽不展示**，为进阶①铺路。
  - **`index.json` 索引清单**：每条记录含
    `{ 源文件绝对路径, 文本镜像路径, 产品线, 产品, 标题, mtime, hash, 媒体资产列表 }`。
    → 这是"来源标注""命中文件溯源""范围检索"的共同地基。
- **增量更新**：按 mtime/hash 比对，只重转改动过的文件。
- **KB 路径可配置**：写进 `settings` store（默认指向上面的 Desktop 路径）。

### 5.2 ②向导层（复用现有 chat，零新增进程）

- **入口**：`ScenarioQuickStart`（侧栏场景卡）新增一张"写方案"卡。
  点击 → 轻量产品选择（从 `index.json` 读产品线/产品列表）。
  **写死的"模板" = 选定产品 + 一套固定章节骨架**
  （背景 → 现状/需求 → 目标/定位 → 总体方案/架构 → 功能 → 价值/成效）。
- **会话 cwd 指向文本镜像目录**：fusion-code 的 `Read/Grep/Glob` 直接翻镜像，
  实现 **agentic 检索**（AI 自己找相关 md 读），无需向量库。
- **章节进度**：章节骨架灌进 `todos` store，左侧侧栏复用为"章节进度条"。
- **"不臆想"系统提示词（写死常量）**，强制三条纪律：
  1. 只用文本镜像里检索到的原文写，不得用模型自身知识补内容；
  2. 每段标注来源文件（据《XX.docx》）；
  3. 知识库查不到的内容，写"⚠️ 资料缺失：<缺什么>"，绝不编造。

### 5.3 ③文档层 + ④导出

- 复用 `WorkspaceTreePanel` 所在的右侧区域，放一个**轻量 Markdown 可编辑预览**
  （不做 Notion 式重型块编辑器）。AI 写完一段落到右侧，用户可直接改字。
- MVP 导出 **Markdown** 文件（落盘到用户选定路径）。
- 导出做成可扩展的格式适配层（Markdown 为第一个适配器），**Word / PDF 等格式留进阶**
  补充，不另起炉灶。

### 5.4 改动落点（贴合现有架构）

| 类别 | 内容 |
|------|------|
| 新增 | `scripts/build-kb-index.ts`；右侧 Markdown 文档面板组件；"方案专家"系统提示词常量 |
| 复用/小改 | `ScenarioQuickStart`（加卡 + 产品选择）；`todos` store（当章节进度）；会话 cwd 指向镜像；`settings`（KB 路径 + 镜像路径）；可能新增"选产品/导出"的 IPC 通道（按 CLAUDE.md 四件套同步改） |
| **不碰** | 进程模型、权限 PermissionBroker、ChatEngine 多 runtime 核心不变量 |

> 注意 CLAUDE.md：加一条 IPC 要同步改四处
> （`ipc-channels.ts` → `preload/index.ts` → `preload/index.d.ts` → main handler），
> 且实际代码在 `apps/desktop/src/`（CLAUDE.md 里的 `src/` 是旧路径）。

### 5.5 MVP 明确不做（YAGNI）

自定义模板编辑器、向量检索/embedding、Word 导出、多产品混合方案、案例库自动推荐——
第一版全不做。先把"选产品 → 对话 → 基于真实资料出草稿 → 导出 md"最小闭环跑通。

## 6. 进阶路线图（未来阶段，MVP 须预留地基）

### 功能① 本地精准检索 + 把图/内容插入方案
- 在指定路径范围检索具体内容（尤其"树状图"这类图片/图表），插到方案光标处。
- 依赖：索引层的**媒体资产抽取**（MVP 已做）+ 右侧编辑区支持图片插入（进阶做 UI）
  + 检索范围可限定到子目录。

### 功能② 联网检索 + 插入
- 用 fusion-code 自带 WebSearch/WebFetch 联网取料，插到方案指定位置。
- ⚠️ 边界：联网内容必须**明确标成"外部来源 + URL"**，与"基于知识库"的内容
  在方案中区分，不得污染"只用知识库"的底线。

### 功能③ 范围检索 + 命中文件溯源
- "在某路径范围找 X，并告诉用户在哪些文件里找到的"——带引用的检索。
- 依赖：`index.json` 的**镜像→源文件路径映射**（MVP 已做）+ 检索范围限定
  + "列出命中文件"交互。

### 进阶三者的共同地基（MVP 必须打对，否则堵路）
> 一个带 **源路径 + 产品线 + 媒体资产** 的 `index.json`
> + 可限定范围的 **agentic 检索**
> + 强制 **来源标注/区分** 机制。

## 7. 主要风险

- **转换质量**：复杂排版的 docx/pptx 转 Markdown 可能丢表格/图结构 → markitdown 主力 +
  LibreOffice 兜底，并在 MVP 抽样人工校验若干成品方案的转换效果。
- **检索召回**：纯 Grep 对"语义相近但措辞不同"的内容召回弱 → MVP 接受（产品线目录
  已大幅缩小范围）；语义检索留进阶。
- **"不臆想"能否真守住**：靠提示词纪律 + 来源标注，需在 MVP 验收时专门测
  "知识库没有的内容 AI 是否老实说缺失"。

## 8. MVP 验收标准

1. 跑一次 `build-kb-index.ts`，3GB KB 生成文本镜像 + 媒体资产 + `index.json`，可增量。
2. 在 App 内点"写方案"卡 → 选"智能导诊" → AI 逐段对话写出草稿，每段带来源标注。
3. 故意要求一段知识库里没有的内容 → AI 标"⚠️ 资料缺失"，不编造。
4. 右侧可改字，导出为 Markdown 文件。
