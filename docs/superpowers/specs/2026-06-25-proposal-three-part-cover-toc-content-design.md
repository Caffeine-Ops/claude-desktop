# 方案三段式生成：封面 → 目录 → 正文

日期：2026-06-25
分支：Install-Plan
状态：设计定稿，待写实现计划

## 背景与目标

「知识库驱动方案写作」当前只覆盖**正文**：AI 用哨兵块（`===方案正文开始===`/`===方案正文结束===`）逐段输出，每个闭合块成一节（`ProposalSection`），在 `ProposalPaper` / `ProposalPreview` 里渲染、可就地编辑、导出 Word/.md。

用户诉求：把一份方案的生成拆成**三个有序阶段**——

1. **封面**：AI 先问关键信息（客户名、方案主题等），再据知识库生成封面草稿；
2. **目录**：AI 提一份建议章节大纲，用户可直接改或口述调整，AI 跟随修订；
3. **正文**：用户确认目录后，AI 按已定稿的目录逐章撰写（即现有能力）。

硬约束：

- 封面、目录、正文三段都要落进右侧方案草稿、都能就地编辑。
- 用户必须**先确认封面、再确认目录**，正文才开始——阶段不可跳步。
- 目录定稿后即「正文的真相源」，正文阶段 AI 对着确认过的目录写；目录确认后不随正文自动变。
- 延续既有底线：**只用知识库、绝不臆想**，查不到标「⚠️ 资料缺失」，分段标来源。

## 选定方案：路线 A —— 阶段隐式标记

复用现有哨兵机制，**不**为封面/目录另立哨兵。引入「当前阶段」状态作为打标签与门控的单一真相源：

- `proposal` store 增 `phase: 'cover' | 'toc' | 'content'`，`start()` 时为 `'cover'`。
- 每个 `ProposalSection` 增 `kind: 'cover' | 'toc' | 'content'`，由「该哨兵块到达时的 `phase`」决定。
- 阶段推进靠草稿面板里的**显式按钮**：按钮先推进 `phase`，再向 AI 发一条推进指令消息。
- 因为只有按钮能推进、一次只有一个阶段活跃，AI 的哨兵输出不会错标。

被否决的路线 B（封面/目录各立一套哨兵）更鲁棒但要扩哨兵机制、加重提示词、AI 写错哨兵就掉块；在「按钮死锁阶段、一次一个阶段」前提下收益不抵成本。

### 为什么阶段推进要靠按钮发的消息，而不是改系统提示词

方案系统提示词（`buildProposalAppend`）在子进程 spawn 时**烘焙、整生命周期不可变**（见 engine.ts 不变量）。它无法「每轮告诉 AI 现在第几阶段」。所以分工是：

- **系统提示词**：描述完整三阶段工作流与各阶段纪律（一次写死、覆盖全程）。
- **按钮发的用户消息**：每轮驱动阶段转换（「封面已确认，请生成目录大纲」/「目录已确认如下：…，请按目录逐章撰写正文」）。
- **store.phase**：UI 显示与给哨兵块打 `kind` 的真相源，由按钮推进。

三者对齐：按钮**先**推进 `phase`、**再**发消息，AI 该轮的哨兵输出按新 `phase` 落到对应区。

## 架构与数据流

```
封面阶段(phase='cover')                目录阶段(phase='toc')              正文阶段(phase='content')
──────────────────────                ──────────────────                ──────────────────────────
AI 问客户名/主题…(无哨兵,留对话)         AI 据知识库+常见结构提建议大纲       AI 按确认目录逐章写
AI 生成封面→哨兵块→kind='cover'         →哨兵块→kind='toc'                每章→哨兵块→kind='content'
用户就地编辑封面                        用户就地改/口述调整,AI 跟随修订       用户就地编辑各章
        │                                      │                                 │
   [确认封面,生成目录]                    [确认目录,开始正文]                  （现有能力）
   phase:cover→toc                       phase:toc→content
   发消息驱动 AI                          发消息(含目录正文)驱动 AI
```

### 各单元职责

**1. `stores/proposal.ts`（状态真相源）**
- 新增 `phase` 状态 + `advancePhase(to)` / 或 `confirmStage()` action。
- `ProposalSection` 增 `kind`。
- `appendSections(messageId, blocks, truncated)` 内部读当前 `phase`，给新节打 `kind`。签名可不变（kind 由 store 内部 phase 决定），降低调用方改动面。
- `start()` / `reset()` 初始化 `phase='cover'`。

**2. `runtime/FusionRuntimeProvider.tsx`（消息进出）**
- `end` 事件里的抽取逻辑**不变**（仍 `extractProposalDraftResult` → `appendSections`）；`kind` 在 store 内按 phase 打，这里无感。
- 抽出一个可复用的「程序化发送」通道，供面板阶段按钮发推进消息：复用现有 send 逻辑（`proposalMode=true` + `proposalProducts=ps.products` + `chatApi.send` + `startAssistantMessage` 预翻转）。当前发送逻辑内嵌在 assistant-ui 的 `onNew` 适配器里，需提取成一个 store/helper 能调的函数（如挂到 chat store 或一个 `sendProposalMessage(text)`）。**这是本设计最主要的集成点与风险点**，实现计划要专门一步处理，确保与 composer 走同一条 send 路径（同样的播种/产品/proposalMode 语义）。

**3. `core/proposalPrompt.ts`（系统提示词）**
- `buildProposalAppend` 改写为描述**三阶段工作流**：
  - 阶段一封面：先问客户名/方案主题/落款等关键信息，信息齐了再生成封面，封面正文用哨兵块包裹。
  - 阶段二目录：据知识库与售前方案常见结构提一份章节大纲（编号列表），用哨兵块包裹；用户会增删改或口述调整，按用户修订重出目录。
  - 阶段三正文：严格按用户**已确认的目录**逐章撰写，章节与顺序以目录为准、不自行增删；沿用现有正文纪律（逐条介绍、资料缺失标注、一次聚焦一部分等）。
- 明确：用户会通过界面按钮发来「确认封面/确认目录」的推进消息；收到后才进入下一阶段，不要自行跳阶段、不要在封面阶段就写正文。
- 哨兵规则（现规则 5）保留不变——三阶段都用同一对哨兵，靠 store.phase 区分 kind。

**4. `components/workspace/ProposalDocPanel.tsx` + `ProposalPaper.tsx`（草稿渲染与阶段控制）**
- 按 `kind` 把 sections 分三区显示：**封面 / 目录 / 正文**，各区带小标题分隔（空区不显示或显占位）。节内编辑/上移/下移/删除沿用现有工具条。
- 顶部加**阶段条**：显示当前阶段（封面 → 目录 → 正文）与对应的推进按钮：
  - `phase='cover'`：按钮「确认封面，生成目录」（封面区为空时禁用）。
  - `phase='toc'`：按钮「确认目录，开始正文」（目录区为空时禁用）。
  - `phase='content'`：无推进按钮（或显「正文撰写中」）。
- 推进按钮：调 `advancePhase` + 程序化发送对应推进消息。「确认目录」那条消息把当前目录区的 markdown 一并带给 AI（目录驱动正文）。

**5. `core/proposalExport.ts` + `proposalDocx.ts`（导出）**
- sections 已按 append 顺序天然是封面→目录→正文（阶段顺序保证），导出顺序无需额外排序。
- docx 增分页：封面单独一页、目录单独一页、正文起新页（在 kind 边界插 page break）。.md 导出维持纯文本拼接、阶段间空行分隔即可。

## 错误处理与边界

- **跨阶段编辑**：封面/目录/正文节始终可就地编辑，不受 phase 限制。「返回上一阶段改」即直接改对应区文本，无需回退 phase。
- **目录确认后想改目录**：用户可直接编辑目录区文本；已生成的正文不自动重写（与「目录确认后不随正文自动变」一致）。如需让 AI 据新目录重写，由用户在对话里另行要求，不在本期自动化。
- **空区禁用门**：封面区为空时「确认封面」禁用，目录区为空时「确认目录」禁用，避免空阶段跳步。
- **截断恢复**：现有 `truncated` 残文恢复逻辑不变，残节按当前 phase 打 kind、带「疑似截断」徽标。
- **warm-spawn**：现有 `proposalMode && !spawnedWithProposal` 注入逻辑（engine.send）不受影响——三阶段共用同一会话同一进程，提示词烘焙/注入时机与现状一致。
- **phase 不持久化**：与现有 store 一样，phase 是会话内内存态；`start()` 重置为 cover。本期不做跨重启恢复（与现状一致）。

## 测试与验证

无单元测试 / 无 ESLint，`bun run typecheck` 是唯一自动门。新增 `phase`/`kind` 字段、新 action、提示词改写、面板分区与按钮都要过 typecheck。

dev 手测路径（CDP 9222 可驱动，见项目记忆）：
1. 开方案会话 → 默认封面阶段 → AI 问关键信息 → 答 → 生成封面落入封面区、可编辑。
2. 点「确认封面，生成目录」→ phase 进 toc → AI 出目录大纲落目录区 → 改/口述调整 → AI 跟随。
3. 点「确认目录，开始正文」→ phase 进 content → AI 按目录逐章写、落正文区。
4. 导出 Word：封面/目录/正文分页、顺序正确、内容与草稿一致。
5. 底线复核：全程内容可溯源到知识库、缺料处标注、AI 不臆想。

## YAGNI / 明确不做

- 不为封面/目录另立哨兵（路线 B）。
- 不做「目录改了自动重写正文」的联动。
- 不做 phase 跨重启持久化。
- 不做封面图片/Logo 上传（封面先为文字版，后续可单列需求）。
- 不做正文章节与目录条目的双向锚点跳转（导出分页足够）。
