# 本地文件知识库（local-kb）

知识库是**两份索引 json**，按文件类型分流（结构完全相同，见下方 schema）：

| 索引 | 收录什么 | 判定 |
|---|---|---|
| `<KB_DIR>/KB-INDEX.json` | 文档（表格/PPT/PDF/markdown/代码…） | 扩展名**不是**图片 |
| `<KB_DIR>/KB-IMAGE-INDEX.json` | 图片 | 扩展名是 png / jpg / jpeg / gif / webp / bmp / heic / svg |

每个已收录文件一条 entry（路径 + 类别 + 一句话概览），逐条 upsert；用户加一个文件，
你就往**对应索引**upsert 一条。**没有"库根文件夹"的概念**——单个任意路径的文件都能加。

`<KB_DIR>` 就是知识库目录（`~/.cowork`），已加进你的可读+可写范围。不确定绝对路径时先跑：
```
node <此skill目录>/scripts/kbpath.mjs
```
它会打印 `kbDir / indexPath / imageIndexPath / categoriesPath / imageCategoriesPath`
五个绝对路径。**绝不把索引写到用户自己的文件夹里。**

**这两份 json 各有两个写手，你是其中之一**：桌面应用的「更新知识库」按钮（文档识别页 /
图片识别页各一个）会全量重扫授权目录并按文件名重新归类（它保留你写过的 summary、但会
重算 category）。所以：写回时必须保持**严格合法 JSON**（一处语法错就会让应用端把整份
索引当损坏处理）。

## 类别（按域各一套，以对应 categories 文件为准，不许自造）

类别集合是**用户可自定义的**（应用「分类管理」页维护，文档/图片两段独立）。挑类别前：

1. 先读对应域的类别文件（kbpath.mjs 打印路径）：文档 → `KB-CATEGORIES.json`，
   图片 → `KB-IMAGE-CATEGORIES.json`。存在就用它的 `categories` 数组（顺序即用户
   偏好，末位恒「其他」）。
2. 文件**不存在**时用对应域的出厂默认（与应用端 `DEFAULT_KB_CATEGORIES` /
   `DEFAULT_KB_IMAGE_CATEGORIES` 同步）：

**文档默认八类：**
```
数据报表   —— 工资表、付款模板、csv 导出、统计台账
财务票据   —— 打款凭证、发票、报销单
合同协议   —— 合同、授权书、委托函、license
汇报演示   —— PPT、pitch、汇报材料
报告方案   —— 审查/审计报告、建设方案、白皮书
学习资料   —— 教程、技术文档、笔记、课件
人事行政   —— 简历、offer、绩效、行政通知
其他       —— 以上都不贴切时的兜底
```

**图片默认七类：**
```
截图       —— Screenshot / CleanShot / 截屏 / 应用界面截图
照片       —— IMG_ / DSC / 相机与手机拍摄
设计素材   —— logo / icon / banner / poster / 切图导出
图表配图   —— chart / diagram / 流程图 / 架构图 / 数据图
证件票据   —— 证件照 / 发票凭证扫描件 / 合同页拍照
表情梗图   —— meme / 表情包
其他       —— 以上都不贴切时的兜底
```

无论哪个来源，`category` 都必须**逐字**取自对应域集合中的一项；拿不准就「其他」。

## 索引 json 格式（两份索引 schema 相同）

```json
{
  "version": 1,
  "updatedAt": 1720500000000,
  "entries": [
    {
      "path": "/Users/me/Desktop/示例表格/员工信息示例.xlsx",
      "name": "员工信息示例.xlsx",
      "ext": "xlsx",
      "category": "数据报表",
      "summary": "员工信息表：姓名/工号/部门/入职日期等基础字段",
      "size": 0,
      "mtimeMs": 0,
      "indexedAt": 1720500000000
    }
  ]
}
```

字段约定：
- `path` 是主键：同 path 已存在 → **替换那一条**（upsert），绝不重复。
- `name` = 文件名（含扩展名）；`ext` = 小写扩展名（不带点）。
- `size` / `mtimeMs` 拿不到就填 `0`（应用端全量重建时会补真值）。
- `indexedAt` / `updatedAt` = 当前 epoch 毫秒（`date +%s` 秒数 ×1000 即可，不必精确）。
- 写回用 2 空格缩进的合法 JSON。

## 场景一：添加一个文件到知识库

用户点文件卡片的「添加到知识库」时，会发来一条带**该文件绝对路径**的请求。你要：

1. **拿到索引路径并定域。** 跑 kbpath.mjs；按扩展名分流——图片扩展名 →
   `imageIndexPath` + 图片类别，其余 → `indexPath` + 文档类别。

2. **读这个文件、提取内容。** 按类型：
   - **文本/代码/markdown/json/csv 等**：直接 `Read`。
   - **图片（png/jpg 等）**：直接 `Read`——你能**看到图像内容**，据画面写概览
     （界面截图写"什么应用的什么界面"、照片写拍了什么、图表写展示什么数据）。
   - **xlsx（Excel 表格）**：它是 zip+xml，用系统自带的 `unzip` 提取文本，别赌
     `require('xlsx')`（那个包在打包后的用户机器上不可用）。**两种编码都要试**——文本
     可能在共享字符串表、也可能内联在 sheet 里：
     ```
     unzip -p "<文件路径>" xl/sharedStrings.xml   2>/dev/null   # 共享字符串（可能不存在）
     unzip -p "<文件路径>" xl/worksheets/sheet1.xml              # 内联字符串在这里
     ```
     两个输出里的 `<t>...</t>`（或 `<t xml:space="preserve">...`）标签就是单元格文本。
     `sharedStrings.xml` 不存在（报 "filename not matched"）很正常——那说明这张表用内联
     字符串，文本全在 sheet1.xml 里。表大只看前若干条足以概括即可。
   - **其它二进制（docx/pdf 等）**：暂不支持内容提取。仍可只凭文件名 + 用户上下文
     写一个保守概览，并在回复里说明"未读取内容，仅凭文件名"。

3. **写一句概览、挑一个类别。** 概览一句话（约 40 字内），抓"内容是什么"。例：
   `员工信息表：姓名/工号/部门/入职日期等基础字段`、`设置页外观分区的界面截图，深色
   主题`。**不要**写"这是一个 xlsx 文件"这种废话。类别从**对应域**的集合里挑最贴切的
   一个，拿不准就「其他」。

4. **upsert 对应索引。** `Read` 现有索引（不存在则从
   `{"version":1,"updatedAt":0,"entries":[]}` 起头）→ 同 path 已在 entries 里就替换那一条、
   不在就 append → 更新顶层 `updatedAt` → `Write` 整份写回（严格合法 JSON，2 空格缩进）。

5. **回报**：告诉用户"已加入知识库"（带类别），并说该索引里现在有多少个文件。

## 场景二：查询（用户日常提问）

用户问「上次那个 X 文件在哪 / 那份关于 Y 的资料怎么说 / 我的截图有多少张 / 找讲 Z 的文档」时：

1. **按查询语义选索引**（kbpath.mjs 拿路径）：
   - 问**图片/截图/照片/logo/图** → 读 `KB-IMAGE-INDEX.json`；
   - 问文档/资料/表格/报告 → 读 `KB-INDEX.json`；
   - 语义不明确（"找一下关于 X 的东西"）→ **两份都读**。
   索引不存在就如实说"该类知识库还是空的"——**别只查了文档索引就断言"没有截图"**，
   图片在另一份索引里。
2. **按 name / summary / category 锁定候选**——匹配用户描述的主题/文件名/关键词，挑最
   相关的 1~3 条。统计类问题（"截图有多少张"）直接按 category/ext 计数作答。
3. **`Read` 打开命中的真实文件**确认内容（图片 Read 能看到画面；xlsx 用 unzip）。索引
   概览只用来定位，**回答必须基于真实文件内容**，别只凭概览就答。
4. **回答并注明来源**：明确说是哪个文件（路径），让用户能去打开核对。

找不到明显匹配就如实说"索引里没有相关文件"，别硬猜。细节见 `reference.md`（按需读）。
