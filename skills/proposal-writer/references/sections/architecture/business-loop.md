# 业务闭环架构

讲系统如何形成业务闭环——从触发到反馈的完整流转，体现"用起来能转起来"。

## 何时用 / 不用

- 用：有明确业务流转的系统（导诊→挂号→就诊、预问诊→报告→回写病历）。
- 不用：无明显闭环的工具型系统。

## 缺失信息优先提问顺序

1. 业务从哪触发、经过哪些环节、到哪结束/反馈
2. 各环节间的数据/动作流转

## 结构骨架（逐行）

- 一段说明业务闭环的整体逻辑
- 一张业务闭环流程图

## 写作要点

- 闭环的环节/步骤**来自素材**，别为"闭环好看"补造环节。
- 强调"闭环"：结果如何回流、驱动持续优化。

## 本节常见呈现变体

- 段落 + 一张 mermaid 流程/闭环图。

## 配图 / 结构图建议

- mermaid flowchart，可用箭头形成回环（见 `methodology/images-and-figures.md`；节点含中文标点用英文双引号包裹）：
```mermaid
flowchart LR
  A["患者发起导诊"] --> B["对话引导"] --> C["科室推荐"] --> D["对接挂号"] --> E["就诊"]
  E --> F["数据回流分析"] --> A
```

## 正例 / 反例

- 正例：闭环各环节都是系统真实支持的功能，箭头流转符合实际业务。
- 反例：硬凑一个"AI 自动诊断→自动开药"的环节（素材没有、且不合规）——编造。
