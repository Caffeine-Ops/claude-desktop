# 系统建设价值

讲系统给客户带来的价值——通常**分若干价值点逐条展开**，是打动决策层的关键节。

## 何时用 / 不用

- 用：几乎所有对客方案。
- 不用：POC 方案可用"预期收益"简化替代。

## 缺失信息优先提问顺序

1. 系统能带来哪几方面价值（素材里的价值主张）
2. 每个价值点的支撑事实/数据（有则更有力）

## 结构骨架（逐行，每价值点一小节）

逐条价值点，每条：**小标题（价值点）+ 一段说明（凭什么带来这个价值）**。医疗范例的价值点如：

- 提升患者就医便利性
- 提升医生门诊效率
- 优化医院门诊资源配置
- 支撑医院智慧服务评级建设（政策价值）
- 促进 AI 应用持续优化

## 写作要点

- **逐条**呈现（`methodology/presentation-modes.md`），别揉成一大段。
- 每个价值点要有"凭什么"——落到具体机制/功能，别只喊口号。
- 涉及量化收益（"提升 X%""节省 Y 分钟"）**必须有素材出处**，无据标缺料，别编数字。

## 本节常见呈现变体

- 默认逐条（小标题 + 说明）。价值点带数据时可配一个对比表。

## 配图 / 结构图建议

- 可配价值/成效对比图表（有数据且有出处时）。

## 正例 / 反例

- 正例：「**提升医生门诊效率**：预问诊报告在就诊前生成并回写病历，医生问诊时间平均缩短……（据《试点报告》）」——有机制、数据有出处。
- 反例：「**行业领先**：本系统全面提升医院各项指标。」——空洞、无支撑、无出处。
