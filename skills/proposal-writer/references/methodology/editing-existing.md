# 改写已有方案（edit）

用户丢来一份已有方案 + 修改诉求时走本流程。核心是**最小改动**，像改代码，不重写整篇。

## 流程

1. **摄入原稿**：读用户给的方案文件（docx/pdf/md，摄入方法见 `SKILL.md` 素材摄入）。
2. **解析现有结构**：抽出它的章节大纲/层级。**edit 不重新选类型**——就在原结构里改。
3. **弄清修改诉求**：常见几类——
   - 局部改某节（措辞/内容更新）
   - 增 / 删章节
   - 换风格 · 受众 · 篇幅
   - 某节逐条化 / 表格化
   - 融入新素材
   - 重排目录
   拿不准就一次问清（见 `asking-vs-proactive.md`）。
4. **最小改动**：**只改用户要动的部分，其余原样保留**。别趁机重写没让你动的章节、别改动无关措辞。
5. **改动仍守接地底线**：新增内容要有素材依据；无据则标 `⚠️ 资料缺失`，**绝不为"改得丰满"编造**。
6. **输出**：默认给改后的完整 markdown；用户只要变更部分时，就只给动过的章节并说明改了什么。

## 为什么最小改动

已有方案往往是用户或同事花时间写好的，大改会破坏他们认可的部分、也难 review。像代码 diff 一样——
**改动越小越可控、越好核对**。

## 反例

- 用户只说"帮我把第 3 章功能改成逐条"，你却顺手重写了背景、调整了架构章——越权，破坏了原稿。
- 用户让"补充安全性内容"，素材里没有安全设计，你编了一段"采用国密算法、等保三级"——编造，违反接地；
  应标：`⚠️ 资料缺失：安全设计的具体方案（等保级别/加密算法/权限模型，原稿与素材均未提供）`。
