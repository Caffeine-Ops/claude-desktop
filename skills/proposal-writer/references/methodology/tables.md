# 结构化数据用表格

源料是结构化数据时，用 GFM markdown 表格组织，别摊成大段散文——表格更直观、保留源料原本结构。

## 何时用表格

- 参数 / 规格清单
- 功能或方案**对比**
- 分项清单（模块清单、角色权限等）
- 实施 / 时间计划
- 报价 / 配置项

## 规矩

- 表头用**源料里的字段名**。
- 单元格**只填知识库/素材查到的真值**；查不到的写「—」，**绝不为凑满表格编造数据**（编表与编文等害）。
- 表格紧接的下一行仍按 `grounding-and-citation.md` 标来源 `（据《文件名》）`。
- 用表 vs 用段由你按源料形态判断；拿不准走 `asking-vs-proactive.md`（问用户），别默认堆成散文。

## 正例

| 输入方式 | 说明 | 是否支持 |
|---|---|---|
| 语音输入 | 接入院内麦克风阵列 | 支持 |
| 文字输入 | 患者手动输入症状 | 支持 |
| 图片输入 | — | — |

（据《智能导诊产品说明》）

——真值来自素材，未提供的"图片输入"一行如实写「—」，不编。

## 反例

- 把上面这种清单摊成："系统支持多种输入方式，包括语音、文字等，语音接入麦克风阵列，文字则……"
  ——信息密度低、结构丢失。
- 为让表格"满"而给"图片输入"编一个"支持，接入 OCR"——编造，违反接地。
